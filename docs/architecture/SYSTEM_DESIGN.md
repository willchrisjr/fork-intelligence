# System Design

## Architectural outcome

Fork Intelligence is a modular monolith with separately runnable processes:

```text
Browser (Next.js)
  | REST + SSE
FastAPI API ---- PostgreSQL
  | enqueue          ^ durable runs, events, evidence, scores
Redis queue          |
  |                   |
Python worker --------+
  | GitHub REST; authenticated GraphQL accelerator
  | safe Git argv
Bare Git network stores
```

- **Web:** Next.js and TypeScript render the public submission flow and the
  evidence workspace. TanStack Query manages server state; TanStack Table owns
  dense accessible tables; Cytoscape is isolated to bounded graph geometry.
- **API:** FastAPI, Pydantic, SQLAlchemy, and Alembic expose a versioned REST API,
  validate inputs, authorize future protected operations, and stream durable
  progress events over server-sent events (SSE).
- **Worker:** the same Python distribution exposes a separate worker entrypoint
  for GitHub discovery, safe Git analysis, scoring, clustering, and exports.
- **PostgreSQL:** source of truth for runs, snapshots, checkpoints, events,
  provenance, evidence, comparisons, scores, classifications, and exports.
- **Redis:** disposable queue and coordination layer. Redis loss may delay work
  but never erases the durable analysis state.
- **Git cache:** one bare store per observed network with namespaced refs; no
  analyzed repository is checked out or executed.

This keeps deployment simple while enforcing module boundaries between
transport, orchestration, GitHub access, Git analysis, methods, persistence, and
exports. A module can become a service later without changing domain contracts.

## External interfaces

The product API is REST. SSE streams ordered progress events and polling remains
a fallback. OpenAPI generated by FastAPI is the contract source and is validated
against frontend types in CI.

GitHub access has two paths:

- anonymous and authenticated REST is the correctness baseline;
- authenticated GraphQL is an optional field-batching accelerator with REST
  fallback and field-level provenance.

No live AI provider is connected in the MVP. Classification, scoring, cluster
membership, and labels are deterministic and versioned.

## Request lifecycle

1. API parses `owner/repository` or an HTTPS GitHub URL into a canonical input.
2. An idempotency key plus normalized request/configuration creates or reuses a
   run and commits an `accepted` progress event.
3. Redis receives only the run ID; PostgreSQL contains durable job state.
4. Worker claims the run, executes checkpointed stages, and commits results and
   the corresponding ordered event in one transaction.
5. The browser consumes SSE or polls, showing partial results as soon as their
   stage is committed.
6. A cancelled run stops at a safe boundary; resume creates a new attempt from
   the last valid checkpoint.

## Boundaries and invariants

- Controllers never run Git or perform long GitHub crawls.
- Workers do not expose HTTP concerns or mutate OpenAPI contracts.
- Domain methods consume immutable versioned metric/evidence inputs.
- Every user-facing conclusion carries evidence IDs, method version, confidence,
  coverage, and calculation time.
- Repository text is untrusted data and cannot influence control flow beyond
  validated parser inputs.
- PostgreSQL rows are the durable truth and Redis is reconstructable. Git stores
  are acceleration caches only until a commit becomes cited evidence; sealed
  evidence objects and refs are retained with the run because force-pushes or
  deletion can make them impossible to fetch again.
- A run reads one sealed snapshot of refs. Concurrent refreshes write temporary
  refs and do not change evidence beneath an active calculation.

## Progressive depth

| Depth      | Scope                                     | Output                                                                                 |
| ---------- | ----------------------------------------- | -------------------------------------------------------------------------------------- |
| Metadata   | Broad accessible network                  | identities, relationships, activity/adoption indicators, preliminary shortlist         |
| Structural | Shortlisted default and priority branches | merge base, ahead/behind, unique commits, stable patch IDs, file composition           |
| Deep       | Bounded high-signal subset                | dependency changes, normalized-diff similarity, clusters, change families, comparisons |

Caps are configuration, not hidden behavior. Results and exports state which
forks/branches reached each depth and why others were excluded.

## Frontend design contract

Implementation follows `docs/DESIGN_CONTRACT.md`: true-white evidence-first
surfaces, compact operational navigation, synchronized inspector and table, and
an accessible table alternative for every graph conclusion. React owns state
and accessibility; Cytoscape owns only bounded geometry/picking in a lazy client
component.

## Failure isolation

Each stage and repository work item has bounded retries. A single unavailable
fork produces an explicit partial failure rather than invalidating the network.
GitHub limit exhaustion checkpoints the run as waiting. Git corruption
quarantines and rebuilds the cache. Exports are generated only from a committed
analysis snapshot.

## Observability

Structured logs include request/run IDs, stage, repository ID, attempt, duration,
and normalized error code, never tokens or raw untrusted content. Metrics cover
queue delay, stage duration, API calls, rate-limit remainder, fetch bytes/time,
cache hits, retries, partial failures, and storage. Health checks distinguish
process liveness from PostgreSQL/Redis readiness.
